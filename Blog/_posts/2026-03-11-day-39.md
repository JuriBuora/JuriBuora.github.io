# 🎯 Goal

The goal of today was to build a **portable and reproducible terminal environment**.

Instead of manually configuring every system separately, the idea was to create a setup that can be deployed quickly on:

- Ubuntu
- Kali Linux
- macOS

This was achieved using a **GitHub dotfiles repository** combined with automated bootstrap scripts.

The core principle:

> My terminal configuration should live in version control and be deployable anywhere.

---

# 🛠 What I Did

## Created a Dotfiles Repository

A **dotfiles repository** stores configuration files used by the shell and other command-line tools.

Typical examples include:

.zshrc  
.p10k.zsh  
.gitconfig  
.vimrc  

These files control:

- shell behaviour  
- prompt configuration  
- plugin loading  
- command aliases  
- development environment settings  

I created a repository called **dotfiles** and pushed it to GitHub so my shell configuration can be **tracked and reused across machines**.

---

## Converted Shell Config Into Symlinks

To allow the shell to use the repository configuration, I replaced the original config files with **symbolic links (filesystem pointers)**.

Resulting structure:

~/.zshrc → ~/dotfiles/.zshrc  
~/.p10k.zsh → ~/dotfiles/.p10k.zsh  

This means:

- the shell still loads the expected files  
- the real configuration lives inside the Git repository  

Any change to the repo automatically affects the shell configuration.

---

## Installed and Standardized Zsh Plugins

Several plugins were installed to improve terminal usability.

### zsh-autosuggestions

Suggests commands based on history.

Example:

typing  
git sta  

suggests  

git status

---

### zsh-syntax-highlighting

Colors commands depending on validity.

Examples:

valid commands → green  
invalid commands → red  

This provides immediate visual feedback when typing commands.

---

### zsh-history-substring-search

Allows history navigation based on partial command input.

Typing:

git

then pressing arrow up cycles through previous git commands.

---

## Installed Powerlevel10k

The shell prompt was customized using **Powerlevel10k**, a fast and highly configurable Zsh theme.

It displays useful contextual information directly in the prompt:

- current directory  
- Git repository status  
- command indicators  

This improves readability and workflow efficiency.

---

## Standardized the Environment Across Machines

The environment was reproduced on:

- Ubuntu VM (source configuration)
- Kali Linux VM
- macOS host

### Ubuntu

Ubuntu was used as the **source-of-truth configuration**.

Steps included:

- creating ~/dotfiles
- copying .zshrc and .p10k.zsh
- initializing Git repository
- pushing configuration to GitHub
- converting local files to symlinks

---

### Kali Linux

Kali initially launched terminals using **bash (Bourne Again Shell)** instead of Zsh.

Steps included:

- installing Zsh
- installing Oh My Zsh
- installing Powerlevel10k
- installing required plugins
- cloning dotfiles repository
- linking configuration files

QTerminal continued launching bash even after changing the default shell.

This was solved by adding the following line to ~/.bashrc:

exec zsh

This forces bash sessions to immediately switch to Zsh.

---

### macOS

macOS already had:

- Zsh
- Oh My Zsh
- Homebrew
- Powerlevel10k

Steps included:

- cloning the dotfiles repository
- installing missing plugin zsh-history-substring-search
- backing up the existing .zshrc
- linking configuration files to the repo

To preserve macOS Ruby tooling, a conditional **rbenv (Ruby version manager)** block was added to the shared .zshrc.

Linux machines ignore it safely.

## Created Bootstrap Setup Scripts

To automate future installations, two setup scripts were created.

### macOS bootstrap

Typical usage:

git clone https://github.com/JuriBuora/dotfiles.git

## Solved Cross-Machine Git Synchronization Issues

Working from multiple machines created Git push conflicts.

Typical error:

push rejected because remote contains work you do not have locally

This occurs when another machine pushed changes first.

Correct workflow:

git pull --rebase origin main  
git push

This avoids unnecessary merge commits and keeps history clean.

# ⚠ Challenges

### GitHub authentication

GitHub no longer accepts account passwords for Git operations.

Authentication must use **Personal Access Tokens (PAT)**.

---

### Divergent Git branches

Working from multiple machines caused branch divergence.

Using `git pull --rebase` resolves this cleanly.

---

### Hidden configuration files

Files beginning with a dot are hidden.

To view them:
